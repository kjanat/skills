# Release notes & release hygiene

How to set release notes easily, plus the rules that keep releases safe.

## The two iron rules (repeat from SKILL.md because they bite hard)

1. **Immutable artifacts.** `plugins.dprint.dev` CDN-caches the release asset. Once a version is published
   and fetched, that binary is frozen — re-uploading a fixed `plugin.wasm`/`plugin.json` to the same
   release will NOT reach users. **A broken release is fixed by bumping the version, never by replacing
   the file.**
2. **Bare semver tags.** Trigger on `*.*.*`, not `v*.*.*`:
   ```yaml
   on:
     push:
       tags: ["[0-9]+.[0-9]+.[0-9]+"] # 0.1.0  — NOT v0.1.0
   ```

### Optional hardening (offer it)

Since artifacts can't be safely mutated, make that explicit at the platform level:

- **Immutable releases**: enable in repo Settings → General (or via a ruleset) so published releases and
  their assets can't be edited/replaced.
- **Block `v*` tags**: add a tag ruleset (Settings → Rules → Rulesets, target `Tags`, pattern `v*`) with
  "Restrict creations" so an accidental `v0.1.0` push is rejected and can't create an off-convention
  release. This keeps the `dprint add <USER>/<NAME>` proxy resolution consistent.

## Approach A — generated `release-fragment.md` (Rust / build.rs)

The build script writes a release body from the schema defaults + the pinned upstream formatter version.
This keeps the install snippet and config example always current. Pattern from tex-fmt's `build.rs`:

````markdown
## Installation

```sh
dprint add <USER>/<NAME>
```

Or add manually to `dprint.json`:

```json
{
  "$schema": "https://dprint.dev/schemas/v0.json",
  "<configKey>": {/* example built from schema defaults */},
  "plugins": [
    "https://plugins.dprint.dev/<USER>/<NAME>-<VERSION>.wasm"
  ]
}
```

## Compatibility

- Based on [<upstream-formatter>](upstream-url) **<upstream-version>**
- Formats files with extensions: `.tex`, `.sty`, …
````

Wire it into the release workflow so GitHub's auto-generated changelog (commits/PRs) is appended below it:

```yaml
- uses: softprops/action-gh-release@v3
  with:
    files: |
      plugin.wasm
      schema.json
    body_path: release-fragment.md
    generate_release_notes: true # appends the autogenerated commit/PR changelog
```

The `build.rs` that emits this (read the upstream version out of `Cargo.lock`, build the example block from
schema property defaults) is fully worked out in tex-fmt's `build.rs` — copy and adjust the extensions and
`<configKey>`.

## Approach B — hand-built body string (process plugin / generated CI)

When the release job is generated (svgo's `.github/workflows/ci.generate.ts`), build the body inline. This
form bakes in the per-version checksum the proxy needs:

````markdown
# <UPSTREAM> <upstream-version>

## Install

Dependencies:

- Install dprint's CLI >= 0.40.0

In a dprint configuration file:

1. Add the plugin (or run `dprint add <USER>/<NAME>`):

   ```jsonc
   {
     "plugins": [
       "https://github.com/<USER>/dprint-plugin-<NAME>/releases/download/<TAG>/plugin.json@<CHECKSUM>",
     ],
   }
   ```

2. Add a `"<configKey>"` configuration property if desired.

   ```jsonc
   {
     "<configKey>": { "pretty": true, "indent": 2 },
   }
   ```
````

In the workflow, the `<TAG>` and `<CHECKSUM>` are filled from step outputs
(`shasum -a 256 plugin.json`). See svgo's `releaseBody()` for the exact interpolation.

## Approach C — native changelog (Go / goreleaser)

shfmt lets goreleaser assemble notes from GitHub's own changelog:

```yaml
# .config/goreleaser.yaml
changelog: { use: github-native }
```

Pair with a hand-maintained `CHANGELOG.md` if you want curated highlights; otherwise github-native +
`generate_release_notes` is enough for a wrapper plugin where most changes are dependency bumps.

## Picking an approach

- **Rust/Wasm** → Approach A (generated fragment + `generate_release_notes`). Lowest effort, always current.
- **Go/Wasm** → Approach C (goreleaser `github-native`), optionally a curated `CHANGELOG.md`.
- **V8 process plugin** → Approach B (you need the checksum-bearing install URL in the body anyway).

In all cases: bump the version for every fix, keep the tag bare-semver, and let the autogenerated
commit/PR changelog carry the routine detail so you only hand-write what's worth highlighting.
